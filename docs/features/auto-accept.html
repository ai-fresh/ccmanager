Auto-Accept

CC Gate can automatically approve Claude Code's tool permission requests, eliminating the need to switch back to your terminal window to type "yes" dozens of times per session. Understand how the 4-tier system works and configure it for your workflow.

What is Auto-Accept?

Every time Claude Code wants to use a tool that requires your approval — reading a file, running a shell command, browsing the web — it pauses and waits for you to confirm. In a long coding session, this can mean dozens or even hundreds of individual approval prompts.

Auto-accept intercepts these prompts before they reach your terminal. When a permission request comes in, the CC Gate hook checks your tier configuration and either approves it immediately (in under a millisecond) or routes it to the CC Gate Questions window for your manual review.

The result is a significantly smoother Claude Code workflow: Claude keeps working without interruption for approved tool types, while requests for higher-risk operations still reach you for explicit approval.

Prerequisite: Auto-accept requires the CC Gate hook to be installed. Go to Settings → Hook → Install to set it up. See the Getting Started guide for full instructions.

The 4 Tiers

CC Gate organizes Claude Code's tools into four tiers based on their potential impact. You can enable each tier independently, giving you precise control over what Claude can do without asking.

1

Read

Risk: Minimal — read-only operations, no side effects

The Read tier covers all operations that only observe your filesystem or the web without making any changes. These operations are entirely safe to auto-approve for the vast majority of use cases — Claude is just looking, not touching anything.

When to enable: Almost always. Enabling Read lets Claude explore your codebase, search files, and look things up on the web without interruption, which dramatically speeds up development sessions.

Read Glob Grep WebSearch WebFetch LS
2

Write

Risk: Low — creates or modifies files, reversible with git

The Write tier covers tools that create or modify files on your filesystem. Changes made by Write-tier tools are fully reversible if you use git — a git diff shows exactly what changed, and git restore undoes it.

When to enable: Enable when you want Claude to make code changes without confirming every file edit individually. Recommended when working in a git repository where you can easily review and revert changes.

Edit Write MultiEdit NotebookEdit
3

Execute

Risk: High — full shell access, commands may be irreversible

The Execute tier covers the Bash tool, which gives Claude Code full terminal command execution on your system. This is the most powerful and risky tier — an approved Bash command can delete files, modify system settings, install packages, make network requests, or run any other terminal command.

When to enable: Only when you have high confidence in both the codebase and Claude's judgment for the current task. Consider enabling it temporarily for specific sessions rather than permanently. Keep it disabled for any session involving unfamiliar codebases or sensitive system directories.

Bash
4

Plan

Risk: Minimal — approves Claude's plan before execution begins

The Plan tier covers ExitPlanMode, which is Claude's request to transition from its planning phase to active execution. When Claude Code uses plan mode, it first outlines what it intends to do and waits for your approval before starting. Enabling this tier means CC Gate approves that transition automatically.

When to enable: Enable when you have reviewed Claude's planning behavior and trust it to proceed without explicit plan approval each time. This tier has minimal risk on its own — the actual execution still requires Read/Write/Execute tier approvals for each individual tool.

ExitPlanMode

Hierarchical Enabling

The 4 tiers are hierarchical: enabling a higher tier automatically enables all tiers below it. This reflects the logical dependency between tiers — you wouldn't want Claude to be able to execute shell commands but not read files.

Plan (4) → also enables tiers 1, 2, 3
Execute (3) → also enables tiers 1, 2
Write (2) → also enables tier 1
Read (1) → standalone, no dependencies

The reverse is also true for disabling: turning off the Read tier disables all tiers, since they all depend on Read being active. Turning off Write disables Write, Execute, and Plan, but leaves Read enabled.

In practice, this means the most common configuration is simply choosing your highest acceptable tier and enabling it — the lower tiers are enabled automatically.

Tip: If you want to quickly revert to "manual everything" mode — for a sensitive refactor session or when working in an unfamiliar codebase — simply disable the Read tier toggle. All tiers are disabled in one click.

Special Cases

Certain tool types are always handled specially regardless of your tier configuration:

💬

AskUserQuestion — Always manual

When Claude Code uses AskUserQuestion, it's asking you a genuine question that requires your input — not just asking permission to use a tool. The CC Gate hook passes these through entirely, allowing Claude Code's native terminal prompt to handle them. You'll see the question appear in your terminal as usual.

❓

Unknown tools — Always manual

If Claude Code introduces a new tool that CC Gate doesn't recognize — either because it's a very new Claude Code feature or a custom extension — the hook will not auto-accept it. The request is routed to the CC Gate Questions window for your manual review. This is a safe default: unknown tools are treated as requiring explicit approval until they are mapped to a tier in a future CC Gate update.

⚡

Safe commands — May bypass hook

Claude Code itself may decide that certain commands are safe enough to run without triggering the PermissionRequest hook at all. Simple read-only commands like echo, ls, and git status may not fire the hook, meaning they are handled by Claude Code internally without ever reaching CC Gate. This is expected behavior — Claude Code applies its own safety judgments before asking for permission.

Session Statistics

The Questions window header shows a counter of how many tool requests have been auto-accepted in the current CC Gate session. This counter resets when CC Gate is restarted.

The statistics give you a quick sense of how much interruption auto-accept is saving you. In an active development session, it's common to see 50–200 auto-accepted Read requests for a single Claude task, depending on the complexity of the codebase exploration.

Individual request details — tool name, project, timestamp, and decision (auto-accepted or manually approved/denied) — are shown in the Questions window list for the current session.

Enabling and Disabling Tiers

Opening the Questions window

  1. Click the CC Gate icon in the menu bar to open the main popup.
  2. Click the bell icon (🔔) in the top-right area of the popup header.
  3. The Questions & Auto-Accept window opens as a floating panel.

Toggling a tier

Each tier has a toggle switch on its row in the Questions window. Click the toggle to enable or disable that tier. Changes take effect immediately — the auto-accept config file is written synchronously and the hook picks up the new setting on the very next permission request.

State persistence

Tier settings are stored in UserDefaults and persist across app restarts. The configuration file at ~/.claude/.ccmanager/auto-accept-config.json is updated every time you change a tier setting, ensuring the hook always has the latest configuration without needing to communicate with the app at runtime.

Technical note: The hook reads the config file directly from disk on every permission request. There is no inter-process communication overhead — the hook decision takes approximately 0ms from the config file read. This design makes auto-accept reliable even when CC Gate's main UI is not focused or the display is off.
