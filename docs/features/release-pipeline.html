Release Pipeline

Automate releases for your own projects — tests, version bump, build, tag, and GitHub Release in one flow.

Overview

CC Gate's Release Pipeline automates releasing your own projects — not CC Gate itself. It detects your project type, runs your test suite, bumps the version number, executes the build, creates a git tag, and optionally publishes a GitHub Release with auto-generated release notes. Everything happens from CommitSheet without touching a terminal.

Supported Project Types Swift (swift build + swift test), JavaScript / Node.js (npm run build + npm test), Python (pytest), C# (.NET build + dotnet test). Custom commands are also supported.

Enabling Release Pipeline

The Release Pipeline is available from CommitSheet. After you commit changes on the main branch, a "Release" button appears in CommitSheet alongside the Push button. Clicking it opens the Release configuration wizard if this is your first release, or proceeds directly to the release flow if already configured.

CI/CD Pipeline Wizard

Introduced in v2.11.12, the CI/CD Pipeline Wizard helps you set up automated releases even if your project has no existing CI configuration.

  1. Detection

    CC Gate scans for existing CI configuration: GitHub Actions YAML files in .github/workflows/, Makefile targets, or shell scripts with known release patterns.

  2. Configuration or Generation

    If no CI is found, CC Gate offers to generate a GitHub Actions YAML workflow tailored to your project type. If CI already exists, it reads the existing configuration and extracts the test and build commands.

  3. Command Setup

    Configure your test command, build command, and version bump strategy. CC Gate shows auto-detected defaults and lets you override each one. Settings are saved per project.

  4. Workflow File Commit

    If CC Gate generated a new GitHub Actions YAML, the WorkflowGenerator creates the file at .github/workflows/release.yml and commits it automatically with a ci: add release workflow commit message.

Release Flow

Once configured, clicking "Release" in CommitSheet triggers the full pipeline in sequence:

1

Tests

Runs your configured test command. Elapsed time is shown live. If tests fail and "Require passing tests" is enabled, the release is aborted and the error output is shown. You can disable this check in configuration to release despite failing tests.

2

Version Bump

Increments the version number in your project's version file (e.g. package.json, AppVersion.swift, pyproject.toml). Choose patch, minor, or major. The new version is shown before confirming.

3

Build

Runs the configured build command. Output is streamed in real time. If the build fails, the release is halted and you can view the full error output.

4

Commit Version Bump

Commits the version bump with a standardized message: chore(release): X.Y.Z. This commit is separate from your feature commits so the release is cleanly identifiable in git history.

5

Tag

Creates an annotated git tag vX.Y.Z on the release commit with a message describing the release.

6

Push

Pushes both the release commit and the tag to origin/main. The push includes --tags so the tag is visible on GitHub immediately.

7

GitHub Release (optional)

If enabled and a GitHub token is configured, CC Gate calls the GitHub API to create a Release entry for the tag with auto-generated release notes based on commits since the previous tag. A link to the Release page is shown when complete.

Configuration Options

Option Default Description
Require passing tests On Abort release if the test command exits with a non-zero code
Test command Auto-detected Custom shell command to run tests (e.g. npm test, pytest, swift test)
Build command Auto-detected Custom shell command to build the project before tagging
Version strategy Patch Which version segment to increment: patch (0.0.X), minor (0.X.0), or major (X.0.0)
Create GitHub Release Off Publish a GitHub Release via API after pushing the tag

Troubleshooting

Test failures blocking release

Fix the failing tests before releasing, or temporarily disable "Require passing tests" in the Release configuration panel. Disabling this is not recommended for production releases, but can be useful for internal tooling or when tests are flaky due to environment issues.

Build errors

CC Gate streams build output in real time. Scroll through the output panel to find the specific error. Fix the build issue in your editor, then trigger a new release from CommitSheet. The version number will not be double-incremented — only one bump per successful release.

Push rejected

The remote has commits that are not in your local branch. Run git pull origin main in terminal, resolve any conflicts, then return to CC Gate and trigger the release again.

Version file not found

CC Gate auto-detects common version file patterns (package.json, *.swift with version constants, pyproject.toml, *.csproj). If your project uses a non-standard location, open the Release wizard and manually specify the path to the version file in the "Version file" field.

Releases are irreversible Once a tag is pushed to a public repository, it is visible to all contributors. If you release a broken version, create a new patch release rather than deleting the tag. Deleting published tags confuses package managers and CI systems that may have already picked up the release.
